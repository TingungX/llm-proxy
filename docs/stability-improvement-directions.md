# LLM Proxy 稳定性优化方向建议

> 目标：在不改变当前代理核心能力的前提下，降低上游波动、配置错误、数据写入竞争、流式链路异常和前端/运维盲区对服务可用性的影响。

## 1. 背景与当前稳定性基线

本项目是一个 FastAPI LLM 代理服务，核心路径由 `/v1/chat/completions`、`/v1/responses`、`/v1/messages` 等路由进入，通过 Pipeline 完成鉴权、模型解析、协议选择/转换、视觉降级、压缩和上游转发；同时提供端点配置、用量、日志、延迟探测等管理 API 与前端控制台。

当前已经具备若干稳定性基础：

- **连接池复用**：`httpx.AsyncClient` 使用全局直接连接池和代理连接池，限制最大连接数、keep-alive 数量与过期时间。
- **请求链路模块化**：OpenAI Chat 与 Responses 路由都通过 Handler Pipeline 串联步骤，便于插入熔断、限流、审计等横切逻辑。
- **统一生命周期任务**：应用启动时初始化 DB 和运行态 State，并启动小时聚合、每日清理任务，退出时关闭 HTTP client。
- **用量与日志体系**：SQLite 中存在明细、小时、日级聚合数据，并提供日志列表、汇总和筛选 API。
- **回归测试覆盖**：测试目录已经覆盖 SSE、协议转换、日志、用量、HTTP client、模型解析、视觉降级等核心模块。

这些能力为后续稳定性优化提供了良好基础；建议优先补齐“异常前移、故障隔离、退化可控、可观测、可验证”五类能力。

## 2. 优先级总览

| 优先级 | 方向 | 主要收益 | 推荐落地周期 |
|---|---|---|---|
| P0 | 配置校验与启动前自检 | 避免坏配置导致启动失败或错误路由 | 1 周内 |
| P0 | 上游超时、重试、熔断与 failover 标准化 | 降低上游抖动对全站影响 | 1-2 周 |
| P0 | SQLite 写入可靠性与后台任务健壮性 | 降低用量/日志写入阻塞、锁冲突和任务静默死亡 | 1-2 周 |
| P1 | 流式响应错误边界与用量补偿 | 防止半路断流、usage 漏记或误记 | 2-3 周 |
| P1 | 观测指标、健康检查与告警 | 更快发现性能退化和配置问题 | 2-4 周 |
| P1 | 管理 API 输入约束与权限边界 | 避免误操作、异常查询拖垮服务 | 2-4 周 |
| P2 | 前端稳定性与回滚体验 | 降低控制台不可用对运维的影响 | 3-5 周 |
| P2 | 压测、混沌测试与发布门禁 | 把稳定性要求固化到 CI/CD | 持续 |

## 3. P0：配置校验与启动前自检

### 现状

`config.json` 由 `load_config()` 直接读取 JSON，运行态 `State` 再构建模型映射、视觉能力、协议、路径、代理开关等派生 map。当前缺少显式 schema 校验、重复 key/别名检查、上游 URL 格式检查、协议组合合法性检查和配置热更新失败回滚说明。

### 风险

- 单个模型配置缺字段、协议拼写错误或路径配置错误，可能在请求期才暴露。
- family routing 指向不存在的模型时，用户看到的是运行时模型解析失败，而不是配置保存/启动时失败。
- API key、endpoint、model routing 之间的配置关系较复杂，缺少“配置变更前测试”会放大误操作影响。

### 建议

1. **引入配置 schema**
   - 使用 Pydantic 或 `jsonschema` 定义 `models`、`family_routing`、`compression`、`error_handling`、端点 settings 的结构。
   - 对 `upstream_protocol` / `upstream_protocols` 使用枚举：`openai/chat-completions`、`openai/responses`、`anthropic` 等。
   - 校验 `api_base` 必须是合法 URL，`upstream_paths` 必须以 `/` 开头。
2. **配置保存前 dry-run**
   - 在 `/api/config` 保存前执行 schema 校验、模型映射构建、routing 目标存在性检查。
   - 校验失败时返回结构化错误，提示字段路径与修复建议。
3. **启动自检报告**
   - 启动时输出模型数量、可用协议、端点数量、默认端点状态、DB path、代理模式等摘要。
   - 对 deprecated 字段继续警告，但增加“是否影响运行”的结论。
4. **热更新回滚**
   - `State.reload()` 先在临时对象中构建派生 map，全部成功后再原子替换当前 state。
   - 保存配置时保留最近 N 个版本，支持控制台一键回滚。

### 验收标准

- 坏配置无法保存；坏配置无法悄悄进入运行态。
- 启动日志能明确显示“配置校验通过/失败”和失败字段路径。
- 增加配置 schema 单测，覆盖协议拼写错误、routing 指向缺失、非法 URL、缺少必填字段。

## 4. P0：上游超时、重试、熔断与 failover 标准化

### 现状

HTTP client 当前使用单一总超时和连接池限制；代理层已有重试状态码和最大重试常量，也存在 endpoint/global 的 failover/no-retry 开关，但不同协议、流式/非流式、同协议/跨协议路径的失败处理语义需要进一步统一。

### 风险

- 单一 120 秒 timeout 难以区分连接超时、读超时、写超时和池等待超时。
- 429/503 重试如果没有指数退避和 jitter，容易在上游限流时形成同步重试风暴。
- 某个上游持续失败时，如果没有熔断和半开探测，请求会持续打向不可用服务。
- failover 后如果没有统一记录，排障时难以知道最终使用了哪个模型和失败原因。

### 建议

1. **拆分 timeout 配置**
   - 按 connect/read/write/pool 设置默认值，并允许模型或 endpoint 覆盖。
   - 流式请求 read timeout 应独立配置，避免长输出被误杀。
2. **标准化 retry 策略**
   - 对 429/503/连接错误采用指数退避 + jitter。
   - 明确非幂等风险：只重试“尚未收到上游响应体”的失败；流式开始后不自动重试。
   - 将 retry 次数、退避耗时、最终状态写入日志和 usage 维度。
3. **增加 circuit breaker**
   - 以 `(endpoint_id, model_id, upstream_protocol)` 为维度统计滑动窗口错误率。
   - 达到阈值后短期熔断，直接 failover 或返回明确错误。
   - 半开状态允许少量探测请求恢复。
4. **统一 failover 语义**
   - 定义“哪些错误可 failover”：429、503、连接失败、上游 overloaded；不对 400/401/403 做 failover。
   - 返回响应或日志中记录 `primary_model`、`final_model`、`failover_reason`。
5. **端点级限流与并发保护**
   - 给 endpoint 增加并发上限和队列长度，保护上游与本服务连接池。
   - 超限时返回明确 429，并在控制台展示当前并发。

### 验收标准

- 网络断开、上游 429、上游 503、流式中断都有独立测试。
- 日志可以还原 retry/failover 过程。
- 压测下连接池不会无限堆积等待。

## 5. P0：SQLite 写入可靠性与后台任务健壮性

### 现状

SQLite 负责 usage 明细、小时聚合、日级聚合和 endpoints 配置。应用启动后创建小时聚合和每日清理协程；日志/用量查询 API 直接访问 DB。

### 风险

- 高并发写 usage/logs 时，SQLite 可能出现锁等待或写入放大。
- 后台聚合任务如果执行时间过长，下一轮调度可能漂移；如果取消逻辑不完整，应用退出时可能留下不一致状态。
- 查询 API 的 limit/offset 缺少上限，异常大查询可能拖慢 DB。
- 数据库路径、WAL、busy timeout、索引和备份策略缺少明确运维约束。

### 建议

1. **启用 SQLite 稳定性 PRAGMA**
   - 建议启动时明确设置 WAL、busy_timeout、foreign_keys、synchronous=NORMAL 或 FULL 的选择依据。
   - 将这些设置写入启动日志，便于确认实际生效。
2. **写入队列化或批量化**
   - 将 usage/log write 从请求路径拆出，放入内存队列 + 后台批量 flush。
   - 队列满时采用可观测降级策略：丢弃低价值明细但保留聚合计数，或同步写入兜底。
3. **聚合任务防重入**
   - 小时/每日任务增加分布式或进程内锁，避免多 worker 启动时重复聚合。
   - 记录每次聚合的开始、结束、扫描行数、耗时和异常。
4. **查询保护**
   - 对日志列表 `limit` 设置硬上限，例如 500 或 1000。
   - 对 since/until 做格式校验和最大时间跨度限制。
   - 给常用筛选字段补齐索引，定期 `EXPLAIN QUERY PLAN` 检查。
5. **备份与恢复演练**
   - 提供 `scripts/backup_db.py` 与恢复文档。
   - 每次 schema migration 前自动备份，失败可回滚。

### 验收标准

- 并发写入压测无 `database is locked` 或有可控重试。
- 日志查询大 limit 会被拒绝或截断。
- 后台任务异常会被指标/日志捕获，不会静默停止。

## 6. P1：流式响应错误边界与用量补偿

### 现状

项目已实现 SSE 处理、Responses/Chat 转换流和 Anthropic/OpenAI 双向流转换。代理层在流式路径中会根据完成事件或估算记录 usage。

### 风险

- 上游开始返回流后中断，客户端可能只看到断流，日志中缺少明确错误事件。
- 不同协议 usage 字段位置不同，转换后容易出现漏记、重复记或估算误差。
- 流式转换代码如果吞掉异常，可能导致请求状态被记录为 success。

### 建议

1. **统一流式状态机**
   - 为每个流式请求维护状态：`created`、`upstream_connected`、`first_token`、`completed`、`client_disconnected`、`upstream_error`。
   - 最终 usage/status 由状态机统一判定，而不是散落在多个 generator 的 `finally` 中。
2. **客户端断开感知**
   - 定期检查 request disconnect，客户端断开后主动关闭上游 stream。
   - usage 状态标记为 `client_cancelled`，避免与上游错误混淆。
3. **SSE 合规兜底**
   - 上游错误发生在流开始前，返回普通 JSON 错误。
   - 流开始后错误，尽量发送协议兼容的 error/done 事件，并记录结构化日志。
4. **usage 补偿机制**
   - 当上游缺 usage 时，记录 `usage_source=estimated`。
   - 当转换链路有原始 usage 和转换 usage 时，保留二者来源，避免重复聚合。

### 验收标准

- 流式正常完成、上游中断、客户端取消、转换异常分别有测试。
- usage 记录包含来源和状态，统计页能排除/区分 estimated 数据。

## 7. P1：观测指标、健康检查与告警

### 现状

项目已有 access log、request id 中间件、日志查询 API 和 latency route，但缺少 Prometheus/OpenTelemetry 级别的指标体系和面向运维的健康检查分层。

### 建议

1. **健康检查分层**
   - `/healthz`：只检查进程是否存活。
   - `/readyz`：检查 DB、配置、State、静态资源目录、后台队列是否可用。
   - `/livez`：检查主事件循环是否阻塞，必要时给容器编排使用。
2. **核心指标**
   - 请求数、错误数、p50/p95/p99 延迟，按 route、endpoint、model、protocol、status 分组。
   - 上游连接失败、retry、failover、熔断次数。
   - DB 写入耗时、锁等待、队列长度、聚合任务耗时。
   - 流式首 token 延迟、流完成率、客户端取消率。
3. **结构化日志**
   - 每条请求日志包含 request_id、endpoint_id、model_id、upstream_model、protocol、retry_count、failover_reason、usage_source。
   - 错误日志保留异常类型和上游状态码，但避免泄露 API key。
4. **告警建议**
   - 5 分钟错误率超过阈值。
   - p95 延迟持续升高。
   - SQLite lock/retry 激增。
   - 后台聚合任务连续失败。
   - 单上游熔断超过阈值。

## 8. P1：管理 API 输入约束与权限边界

### 风险

控制台 API 可以影响配置、端点、日志查询和用量统计。如果输入约束不足，容易出现误操作或重查询拖慢服务。

### 建议

- 为所有管理 API 定义请求/响应模型，替代裸 dict 和手动 query parsing。
- 对分页、时间范围、排序字段、枚举字段设置白名单和上限。
- 对配置修改、端点删除、API key 查看/重置增加二次确认或审计日志。
- 前端不展示完整 secret；后端也不在日志和错误响应中输出 secret。
- 对管理 API 与代理 API 使用不同 auth scope，避免业务调用 key 拥有管理权限。

## 9. P2：前端稳定性与回滚体验

### 建议

- 前端 API client 统一处理超时、重试、错误 toast 和 request id 展示。
- 配置编辑页增加“验证配置”“保存草稿”“查看 diff”“回滚到上一版”。
- 日志页/用量页对大数据查询增加 loading cancellation 和分页上限提示。
- 增加端到端 smoke：打开控制台、加载配置、加载日志、切换 endpoint、保存无效配置被拒绝。
- 静态资源构建后增加 hash 校验，容器启动时 readyz 检查 `static/dist` 是否存在。

## 10. P2：压测、混沌测试与发布门禁

### 建议

1. **压测场景**
   - 非流式短请求高并发。
   - 流式长输出并发。
   - 上游 429/503 注入。
   - SQLite 写入高峰 + 日志查询并发。
2. **混沌场景**
   - 上游连接超时、读超时、半路断流。
   - DB 临时锁住。
   - 配置文件损坏或缺字段。
   - 客户端提前断开连接。
3. **发布门禁**
   - 后端 `pytest` 全量通过。
   - 前端 `npm test` / `npm run build` 通过。
   - smoke test 通过。
   - 配置 schema 校验通过。
   - 关键路径压测指标未退化超过阈值。

## 11. 推荐落地路线图

### 第 1 阶段：先阻断明显事故源（P0）

- 配置 schema + 保存前 dry-run。
- HTTP timeout 拆分 + retry 退避 + failover 日志。
- 日志查询 limit 上限 + 时间范围校验。
- SQLite WAL/busy_timeout + 常用索引检查。

### 第 2 阶段：提升故障隔离能力（P0/P1）

- circuit breaker。
- 端点级并发限制。
- 用量/日志写入队列化。
- 后台任务指标与防重入。

### 第 3 阶段：提升可观测和发布质量（P1/P2）

- Prometheus/OpenTelemetry 指标。
- 分层 health/readiness。
- 流式状态机与 usage_source。
- 压测和混沌测试进入 CI 或预发布流程。

### 第 4 阶段：提升运维体验（P2）

- 控制台配置 diff、验证、回滚。
- 日志页 request id 串联上游错误、retry、failover。
- DB 备份恢复脚本与演练文档。

## 12. 自审

### 覆盖性检查

- [x] 覆盖代理核心链路：鉴权、模型解析、协议转换、上游转发、流式响应。
- [x] 覆盖数据链路：usage/logs 写入、聚合、清理、查询保护。
- [x] 覆盖运维链路：健康检查、指标、告警、备份恢复、发布门禁。
- [x] 覆盖前端控制台对稳定性的影响：配置验证、错误展示、回滚。

### 风险排序检查

- [x] P0 优先处理“坏配置、上游抖动、DB 锁/后台任务”这类最容易造成线上不可用的问题。
- [x] P1 处理“可观测、流式边界、管理 API 约束”这类会放大排障成本或引发局部故障的问题。
- [x] P2 处理“前端体验、压测混沌、发布门禁”这类持续改进项。

### 可执行性检查

- [x] 每个 P0/P1 方向都给出了验收标准，可拆成具体 issue。
- [x] 建议尽量基于现有 Pipeline、State、DB、测试体系扩展，而不是推倒重写。
- [x] 没有要求立即替换 SQLite；先给出 WAL、busy timeout、队列化、索引、备份等渐进方案。

### 可能遗漏与后续补充

- 多进程部署模式下，后台任务和内存熔断状态需要额外设计共享锁或外部存储；本文只提出风险和方向，没有指定最终实现。
- 如果未来请求量显著增长，SQLite 可能需要迁移到 Postgres；本文将其作为中长期演进，而不是当前 P0。
- 本文没有给出具体指标阈值，建议在压测和线上基线采集后补充。
